## 快速启动

### 环境要求

- Node.js >= 18
- MiniMax 开放平台账号（获取 API Key）

### 第一步：安装依赖

```bash
cd AI.Product.Management.Assistant
npm install
```

### 第二步：配置 API Key

```bash
cp .env.example .env
```

编辑 `.env` 文件：

```env
MINIMAX_API_KEY=你的 API Key
MINIMAX_GROUP_ID=你的 Group ID   # 可选
PORT=3000
```

> API Key 申请地址：MiniMax 开放平台 → 控制台 → API 密钥

### 第三步：启动服务

```bash
# 正式运行
npm start

# 开发模式（代码变更自动重启）
npm run dev
```

启动成功后终端显示：

```
🚀 AI 助手系统已启动
   Web 控制台: http://localhost:3000
   Webhook:    http://localhost:3000/webhook
```

### 第四步：打开控制台

浏览器访问 → `http://localhost:3000`

## 企业微信群推送（V1.1）

AI 生成结果后，一键将内容推送到企业微信群，团队所有人实时可见。

### 工作原理

```
控制台生成 AI 结果
        │
        ▼
点击「↗ 推送到群」按钮
        │
        │ POST 群机器人 Webhook URL
        ▼
  企业微信群（所有成员可见）
  显示 Markdown 格式内容
```

> **与自建应用的区别：** 群机器人是单向推送（服务器→群），无法接收群内指令触发 AI。适合「生成完主动通知团队」的场景。双向交互请见[高阶版本](#高阶版本企业微信自建应用接入)。

---

### 第一步：创建群机器人

1. 打开企业微信，进入目标群聊
2. 点击右上角「`···`」→ 「添加机器人」→「新建机器人」
3. 填写机器人名称（如：AI 产品助手）
4. 点击「添加」，复制生成的 **Webhook 地址**

> 无需管理员权限，群内任何成员均可操作。

---

### 第二步：填入配置

编辑项目根目录的 `.env` 文件：

```env
WECHAT_WEBHOOK_URL=https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
```

保存后重启服务：`npm start`

---

### 第三步：使用推送

1. 在控制台生成任意 AI 结果（PRD / 数据分析 / 技术评审）
2. 结果区域下方出现「**↗ 推送到群**」按钮
3. 点击后，内容以 Markdown 格式发送到群

推送成功后群内消息示例：

```
📋 AI 产品助手 — PRD 文档

## 用户列表批量操作功能 PRD

### 背景
当前用户列表仅支持逐条操作，运营效率低下…

### 功能列表
1. 全选 / 反选
2. 批量禁用 / 启用
3. 批量导出
…

---
> 由 AI 产品助手自动生成
```

---

### 推送内容说明

| 场景 | 行为 |
|------|------|
| 内容 ≤ 3800 字符 | 完整推送 |
| 内容 > 3800 字符 | 截断推送，并附提示「完整版请下载 .md 文件」 |
| 未配置 Webhook URL | 「↗ 推送到群」按钮不显示 |

## 功能详解与使用场景

### 1. PRD 生成

**适用场景：** 快速将口头需求、会议记录、老板一句话转化为结构化产品需求文档。

**AI 角色设定：** 资深产品经理，输出包含背景、目标、功能列表、验收标准。

---

**场景一：从一句话生成 PRD**

```
输入：
用户需要一个扫码签到功能，适用于线下活动，要能导出签到名单

输出示例：
# 扫码签到功能 PRD

## 背景
当前线下活动签到依赖手工登记，效率低、易出错，需要数字化解决方案。

## 目标
- 提升线下活动签到效率至 <30 秒/人
- 支持实时统计和名单导出

## 功能列表
1. 活动管理
   - 创建活动（名称、时间、地点、预计人数）
   - 生成活动专属二维码
2. 扫码签到
   - 微信/App 扫码，自动核对报名信息
   - 重复扫码提示
   - 未报名人员提示
3. 数据导出
   - 导出 Excel：姓名、手机、签到时间
   - 支持筛选已签到 / 未签到

## 验收标准
- [ ] 扫码响应时间 < 2 秒
- [ ] 支持 500 人同时签到不卡顿
- [ ] 导出文件格式符合 Excel 2007+
```

```
输出示例：
# 会员积分系统 PRD

## 背景
...（完整 PRD 内容）
```

### 2. 数据分析

**适用场景：** 输入数据指标、用户反馈、业务数字，获取分析结论和行动建议。

**AI 角色设定：** 数据分析师，输出结论、趋势判断、改进建议。

---

**场景一：分析核心指标变化**

```
输入：
上周数据：DAU 12万，新增用户 8000，次日留存 34%，7日留存 18%
本周数据：DAU 10.5万，新增用户 6500，次日留存 31%，7日留存 15%
请分析问题

输出示例：
## 数据分析报告

### 核心问题
整体呈下滑趋势，DAU 下降 12.5%，留存率同步恶化，需优先排查留存断层。

### 指标拆解
| 指标 | 上周 | 本周 | 变化 |
|------|------|------|------|
| DAU  | 12万 | 10.5万 | -12.5% ↓ |
| 新增 | 8000 | 6500 | -18.75% ↓ |
| 次留 | 34% | 31% | -3pp ↓ |
| 7留  | 18% | 15% | -3pp ↓ |

### 可能原因
1. 新增质量下降（次留降幅更明显）
2. 产品某功能上线引发负面体验
3. 竞品抢量或外部事件影响

### 建议行动
1. 立即：查本周新增渠道构成，对比留存差异
2. 短期：排查本周是否有功能上线，对比前后留存
3. 持续：建立留存漏斗监控，按用户分群追踪
```

### 3. 技术评审

**适用场景：** 技术方案讨论前快速获取外部视角，识别风险点，补充遗漏考量。

**AI 角色设定：** 技术架构师，输出优缺点分析和改进建议。

---

**场景一：评审技术选型**

```
输入：
我们要做一个实时聊天功能，考虑用 WebSocket 长连接方案：
- 服务端用 Node.js + Socket.io
- 消息存储用 Redis（仅存最近100条）
- 历史消息走 MySQL
- 预计同时在线 1 万人
请评审这个方案

输出示例：
## 技术方案评审

### 方案概述
WebSocket + Socket.io + Redis 热数据 + MySQL 冷数据，标准实时通信分层架构。

### 优点
1. Socket.io 兼容性好，自动降级到轮询，适合快速上线
2. Redis 缓存最近消息，读取延迟低
3. 职责分层清晰，后期易于扩展

### 风险与问题
1. **单点风险**：Socket.io 单节点上限约 5-8 万连接，1 万在线接近瓶颈
   → 建议提前规划多节点 + Redis Adapter（socket.io-redis）
2. **消息丢失**：Redis 仅存 100 条，若服务重启，未落盘消息会丢失
   → 建议异步写入 MySQL，消息确认 ACK 机制
3. **MySQL 热点**：历史消息高并发读取可能打穿 MySQL
   → 建议增加查询缓存层，或评估 TiDB / ClickHouse

### 改进建议
- 短期：加入消息 ACK 确认，防止丢消息
- 中期：接入 socket.io-redis，支持多实例部署
- 长期：考虑将 MySQL 替换为更适合消息场景的时序存储

### 结论
方案可行，建议在上线前完成 Redis Adapter 接入，避免未来扩容时大改。
```

## 配置文件说明

`config.json` 由 Web 控制台实时读写，无需手动编辑。

```json
{
  "features": {
    "prd": true,        // PRD 生成功能开关
    "analyze": true,    // 数据分析功能开关
    "review": true      // 技术评审功能开关
  },
  "model": {
    "maxclaw": true     // MaxClaw (MiniMax) 模型开关
  }
}
```

**说明：**
- 所有开关为 `true` 时功能启用，`false` 时拒绝相关请求
- 关闭某功能后，Webhook 收到该指令会记录日志但不调用 AI
- 关闭 `maxclaw` 后，所有 AI 调用（包括控制台测试）均被拦截

**响应：**
```json

```

---

获取当前配置。

---

更新配置（Web 控制台开关触发）。

**请求体：**

```json
{

}
```

```json
{ "error": "功能 prd 已关闭" }
```

```json
{
  "event": {
    "text": { "content": "/prd 用户登录需求" }
  }
}
```

**响应：** 立即返回 `{"errcode":0,"errmsg":"ok"}`，AI 处理异步进行。

---

检测群机器人是否已配置。

**响应：**
```json

```

```json

  "type": "prd",

```

## 常见问题

**Q: 启动后提示「未配置 MINIMAX_API_KEY」**

复制 `.env.example` 为 `.env` 并填入真实 API Key，重启服务。

---

**Q: 控制台发送请求返回「功能 xxx 已关闭」**

检查左侧功能开关是否已打开，或直接检查 `config.json` 中对应字段是否为 `true`。

---

**Q: Webhook 收到消息但没有触发 AI**

按以下顺序排查：
1. 系统是否处于「运行中」状态（右上角徽标为绿色）
2. MaxClaw 模型开关是否打开
3. 对应功能开关是否打开
4. 消息格式是否以 `/prd`、`/analyze`、`/review` 开头
5. 查看日志区域是否有对应的日志条目

---

**Q: 本地开发如何让企业微信调用到 Webhook？**

使用 ngrok 内网穿透：
```bash
# 安装 ngrok 后
ngrok http 3000
# 将输出的 https://xxxx.ngrok-free.app 填入企业微信 Webhook URL
```

---

**Q: 日志中输入/输出被截断了**

日志截断是设计行为（输入限 80 字符，输出限 120 字符），避免控制台显示过长。完整内容已通过 API 返回给调用方。

---

**Q: 如何修改 AI 的回复风格？**

编辑 `server.js` 中的 `getSystemPrompt` 函数，修改对应功能的 system prompt 即可：

```js
function getSystemPrompt(type) {
  const prompts = {
    prd: '你是资深产品经理，...',     // 修改这里
    analyze: '你是数据分析师，...',   // 修改这里
    review: '你是技术架构师，...',    // 修改这里
  };
  return prompts[type];
}
```

## 高阶版本：企业微信自建应用接入

> 当前 V1.x 使用群机器人单向推送。如需实现「在企业微信中发 `/prd xxx` 指令触发 AI 并回复」的双向交互，需要申请自建应用，作为后续版本规划。

### 自建应用 vs 群机器人对比

| 能力 | 群机器人（V1.x 当前方案） | 自建应用（V2.0 规划） |
|------|:---:|:---:|
| 推送消息到群 | ✓ | ✓ |
| 接收用户指令触发 AI | ✗ | ✓ |
| 指定人接收消息 | ✗ | ✓ |
| 需要管理员权限 | ✗ | ✓ |
| 配置复杂度 | 低 | 中 |

### V2.0 自建应用配置步骤（预留）

**前提条件：** 需要企业微信管理员账号，或由管理员授权「应用管理」子权限。

1. 登录企业微信管理后台 `work.weixin.qq.com`
2. 应用管理 → 创建应用，填写名称、图标、可见范围
3. 进入应用详情 → 「接收消息」→ 设置 API 接收：
   - URL：`https://你的服务器域名/webhook`
   - Token 和 EncodingAESKey：填入 `.env` 对应字段
4. 获取应用的 `AgentId` 和 `Secret`，填入 `.env`
5. 在群聊或单聊中发送指令即可触发 AI

**本地开发时的内网穿透：**
```bash
# 使用 ngrok 将本地端口暴露到公网
ngrok http 3000
# 将生成的 https://xxxx.ngrok-free.app 填入企业微信应用的接收消息 URL
```

**V2.0 新增的 `.env` 字段（规划）：**
```env
WECHAT_CORP_ID=        # 企业 ID
WECHAT_AGENT_ID=       # 应用 AgentId
WECHAT_SECRET=         # 应用 Secret
WECHAT_TOKEN=          # 接收消息 Token
WECHAT_AES_KEY=        # 消息加解密 Key
```
